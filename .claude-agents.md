## 🤖 Agent Modes & Specializations

### Mode 1: Full-Stack Development Agent
**Use When**: Building complete features end-to-end (backend + frontend)

**Capabilities**:
- [ ] Analyze requirements across full stack
- [ ] Design database schema
- [ ] Create backend API endpoints
- [ ] Build frontend components
- [ ] Write comprehensive tests
- [ ] Create Git commit with matching changes

**Instructions**:
1. Start with database schema if data model changes
2. Implement backend service layer first
3. Create API endpoints with validation
4. Build frontend components matching design
5. Write tests for both layers (backend ≥ 80%, frontend ≥ 70%)
6. Create focused Git commit with all changes
7. Run verification: backend tests + frontend build

**Success Criteria**:
- ✅ Database migrations run successfully
- ✅ Backend API returns correct responses
- ✅ Frontend components render without errors
- ✅ All tests pass
- ✅ TypeScript/Python type checking passes

**Example Prompt**:
```
Build a "Project Reviews" feature:
- Freelancers can submit reviews for completed projects
- Clients can submit reviews for freelancer work
- Reviews show rating (1-5 stars) and comment
- Include approval workflow for sensitive content
- Add tests for complete flow
```

### Mode 2: Backend API Specialist
**Use When**: Building/fixing backend APIs, db migrations, services

**Capabilities**:
- [ ] Design database schemas with migrations
- [ ] Write FastAPI endpoints with validation
- [ ] Optimize database queries
- [ ] Implement caching strategies
- [ ] Handle errors gracefully
- [ ] Write comprehensive integration tests

**Instructions**:
1. Always use Turso database (remote only)
2. Write Pydantic schemas first (validation contracts)
3. Use SQLAlchemy ORM (parameterized queries)
4. Implement service layer for business logic
5. Add error handling with proper HTTP status codes
6. Write tests with fixtures for database setup
7. Document endpoints with docstrings

**Code Checklist**:
- [ ] Type hints on all functions
- [ ] Pydantic validation on inputs
- [ ] Async/await for all I/O
- [ ] Proper error handling with try-catch
- [ ] JWT token verification
- [ ] Rate limiting decorator
- [ ] Comprehensive docstring
- [ ] Tests with ≥80% coverage

**Example Prompt**:
```
Create payment webhook handler for Stripe:
- Verify Stripe signature
- Handle charge.succeeded, charge.failed events
- Update payment status in database
- Send notifications to user
- Retry logic for failed updates
- Include error logging
- Write integration tests
```

### Mode 3: Frontend UI/Component Specialist
**Use When**: Building React components, fixing UI, styling

**Capabilities**:
- [ ] Design components with TypeScript
- [ ] Create responsive layouts
- [ ] Implement theme support (light/dark)
- [ ] Add accessibility features
- [ ] Write component tests
- [ ] Optimize performance

**Instructions**:
1. Always use TypeScript with strict mode
2. Create CSS modules per component (.common, .light, .dark)
3. Use Tailwind + CSS modules (not inline styles)
4. Add ARIA labels and semantic HTML
5. Make components responsive (mobile-first)
6. Write Jest tests with ≥70% coverage
7. Use Framer Motion only for complex animations

**Component Checklist**:
- [ ] TypeScript interface for props
- [ ] Default exports only for routes
- [ ] CSS modules with theme support
- [ ] 'use client' directive if interactive
- [ ] ARIA labels on interactive elements
- [ ] Responsive CSS (320px, 640px, 1280px breakpoints)
- [ ] Tests for main functionality
- [ ] Storybook story (optional)

**Example Prompt**:
```
Build ProjectFilter component:
- Filter by status (open, in_progress, completed)
- Date range picker for deadline
- Budget range slider ($100-$100k)
- Search by title/description
- Save filter preferences
- Reset filter button
- Mobile responsive
- Include Jest tests
- Match existing Card styling
```

### Mode 4: Testing & QA Agent
**Use When**: Writing tests, improving coverage, E2E testing

**Capabilities**:
- [ ] Write unit tests (Jest, Pytest)
- [ ] Write integration tests
- [ ] Write E2E tests (Playwright)
- [ ] Analyze coverage gaps
- [ ] Create test utilities and fixtures
- [ ] Verify critical workflows

**Instructions**:
1. Target ≥80% coverage for critical paths
2. Test happy path + edge cases + error cases
3. Use meaningful test names describing behavior
4. Create reusable fixtures for common setup
5. Mock external dependencies (API, email, storage)
6. Run tests in CI/CD pipeline
7. Document test strategy

**Test Checklist**:
- [ ] Happy path tests
- [ ] Edge case tests (empty, null, large values)
- [ ] Error condition tests
- [ ] Integration tests with database
- [ ] Async operation tests
- [ ] Accessibility tests (frontend)
- [ ] Performance tests (if applicable)
- [ ] Fixtures for common setup

**Test Coverage Targets**:
| Module | Target |
|--------|--------|
| Authentication | 90% |
| Payments | 90% |
| Projects (CRUD) | 85% |
| Messages | 80% |
| Profiles | 75% |
| UI Components | 70% |

**Example Prompt**:
```
Write comprehensive tests for payment workflow:
- Test successful payment creation
- Test payment with insufficient funds
- Test webhook signature verification
- Test concurrent payment attempts
- Test payment cancellation
- Test refund process
- Target 90% coverage
- Include fixtures for test data
```

### Mode 6: Security & Optimization Agent
**Use When**: Security audits, performance tuning, vulnerability fixes

**Capabilities**:
- [ ] Audit code for security issues
- [ ] Check dependency vulnerabilities
- [ ] Optimize database queries
- [ ] Improve frontend performance
- [ ] Add security headers
- [ ] Implement rate limiting

**Instructions**:
1. Check OWASP Top 10 vulnerabilities
2. Verify input validation everywhere
3. Check SQL injection risks (ORM usage)
4. Verify XSS prevention (sanitization)
5. Check CSRF tokens on state-changing requests
6. Verify secure headers (HSTS, CSP)
7. Audit dependency versions

**Security Checklist**:
- [ ] Input validation (backend + frontend)
- [ ] XSS prevention (DOMPurify, sanitize HTML)
- [ ] CSRF protection (tokens on POST/PUT/DELETE)
- [ ] SQL injection prevention (ORM only)
- [ ] Authentication properly implemented
- [ ] Authorization checks on sensitive endpoints
- [ ] Sensitive data not logged
- [ ] Rate limiting enabled
- [ ] HTTPS enforced
- [ ] Dependencies up-to-date
- [ ] No hardcoded secrets

**Performance Targets**:
| Metric | Target |
|--------|--------|
| API Response Time | <200ms |
| First Contentful Paint (FCP) | <2s |
| Lighthouse Score | ≥80 |
| Bundle Size | <200KB (gzipped) |
| Database Query Time | <100ms |

**Example Prompt**:
```
Security audit for payment endpoints:
- Check input validation
- Verify JWT token validation
- Check rate limiting
- Verify secure headers
- Check for sensitive data leaks in logs
- Test SQL injection scenarios
- Check CSRF tokens
- Verify user authorization (can't access others' data)
- Generate security report with findings
```

## 🚀 Agent Execution Workflow

1. **Initial Analysis** (5 min)
   - Understand requirement
   - Check existing patterns
   - Identify dependencies

2. **Design Phase** (10-15 min)
   - Create architecture
   - Plan database changes
   - List components/endpoints

3. **Implementation** (30-60 min)
   - Write code following patterns
   - Add tests incrementally
   - Verify type safety

4. **Testing & Verification** (15-20 min)
   - Run unit tests
   - Run integration tests
   - Build verification
   - Manual testing if needed

5. **Documentation & Commit** (5-10 min)
   - Update docs
   - Create Git commit
   - Add comments if needed

## 📞 Escalation Paths

### When to Pause & Ask for Clarification
- ❓ Ambiguous requirements
- ❓ Breaking changes needed
- ❓ Design decisions with trade-offs
- ❓ Complex business logic
- ❓ Security implications
- ❓ Production deployment

### When to Suggest Alternatives
- 📌 Multiple valid approaches
- 📌 Performance vs. simplicity trade-off
- 📌 Tech stack decisions
- 📌 API design choices
- 📌 Database schema options
